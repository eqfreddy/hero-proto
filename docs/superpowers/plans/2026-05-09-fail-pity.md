# Fail Pity Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** After 3 consecutive losses on a `(stage, tier)`, the next attempt fights enemies with -10% HP, one-shot only. Counter resets on any win. Hidden mechanic — never shown as a counter.

**Architecture:** A new `Account.stage_pity_json` JSON column tracks `{(stage_code:tier): consecutive_loss_count, (stage_code:tier:_consumed): bool}`. A new `app/fail_pity.py` module owns the read/update logic. The battle-start path reads pity state and multiplies enemy `max_hp`/`hp` by 0.9× when discount is owed; the battle-end path clears the counter on WIN, increments on LOSS, or resets on LOSS-after-consumed-discount.

**Tech Stack:** Python 3.13 + FastAPI + SQLAlchemy 2.0 + Alembic. Tests via `uv run pytest`. Additive Alembic migration adds one Text column.

**Spec:** `docs/superpowers/specs/2026-05-09-fail-pity-design.md`

**Reference (grounded codebase facts):**
- `app/models.py:191` — `Account` model location
- `app/models.py:216` — `Account.stages_cleared_json` (existing JSON Text column — pattern to mirror)
- `app/routers/battles.py:122-138` — current power-floor check, where pity-start logic will sit alongside (subsystem #2 already shipped)
- `app/routers/battles.py:175-198` — wave loop where enemy `team_b` units are built (HP multiplier applied here)
- `app/routers/battles.py:204-211` — outcome propagation (final `outcome` known after wave loop)
- `app/routers/battles.py:215` — first WIN-only side-effect call (`mark_cleared`); pity-end logic goes adjacent
- `app/tiers.py` — existing module for tier-keyed policy (subsystem #2 home)
- `app/combat.py:CombatUnit` — has `max_hp: int` and `hp: int` mutable attributes
- `alembic/versions/c4f7d8a3e2b9_add_tower_of_trials_columns_to_accounts.py` — recent migration template (Text column on accounts)

**Key spec mechanics (locked):**

| Outcome on attempt N | Pity action |
|---|---|
| WIN | Reset both `[stage_key]` and `[stage_key+":_consumed"]` to absent (delete keys). |
| LOSS, `_consumed=False`, count<3 | Increment count by 1. |
| LOSS, `_consumed=False`, count≥3 | Discount applies on this attempt; mark `_consumed=True`. (See start-of-battle logic below.) |
| LOSS, `_consumed=True` | Reset count to 0, clear `_consumed`. |

The discount is applied at **battle start** when count ≥ 3 and `_consumed` is False; that same start-of-battle path sets `_consumed=True`. So the LOSS-after-consumed branch corresponds to "the discount didn't save them."

```python
"""Fail-pity engine.

After 3 consecutive losses on a (stage, tier), the next attempt fights with
-10% enemy HP. Counter resets on any win. Hidden mechanic — never shown.

State lives in Account.stage_pity_json:
  {
    "<stage_code>:<TIER>": int,            # consecutive-loss count
    "<stage_code>:<TIER>:_consumed": bool, # discount was applied this cycle
  }

The three exposed entry points are pure read/mutate functions on the
Account row. Caller is responsible for the DB commit; we never commit
here so battle-resolve can stage all its writes in one transaction.
"""
from __future__ import annotations

import json
import logging

from app.models import Account, StageDifficulty

log = logging.getLogger(__name__)

PITY_LOSS_THRESHOLD = 3   # losses before discount kicks in
PITY_HP_MULT = 0.9        # enemy HP multiplier on the discounted attempt


def _key(stage_code: str, tier: StageDifficulty | str) -> str:
    tier_str = tier.value if isinstance(tier, StageDifficulty) else str(tier)
    return f"{stage_code}:{tier_str}"


def _consumed_key(stage_code: str, tier: StageDifficulty | str) -> str:
    return _key(stage_code, tier) + ":_consumed"


def _load(account: Account) -> dict:
    try:
        return json.loads(account.stage_pity_json or "{}")
    except (json.JSONDecodeError, TypeError):
        log.warning("stage_pity_json corrupt for account=%s; resetting", account.id)
        return {}


def _save(account: Account, data: dict) -> None:
    account.stage_pity_json = json.dumps(data)


def read_pity(account: Account, stage_code: str, tier: StageDifficulty | str) -> tuple[int, bool]:
    """Return (count, consumed) for a (stage, tier) pity entry. Defaults (0, False)."""
    data = _load(account)
    count = int(data.get(_key(stage_code, tier), 0))
    consumed = bool(data.get(_consumed_key(stage_code, tier), False))
    return count, consumed


def apply_battle_start(account: Account, stage_code: str, tier: StageDifficulty | str) -> float:
    """Called before the wave loop runs. Returns the enemy-HP multiplier
    (1.0 for no discount, PITY_HP_MULT for discounted). Sets _consumed=True
    on the account row when a discount fires."""
    count, consumed = read_pity(account, stage_code, tier)
    if count >= PITY_LOSS_THRESHOLD and not consumed:
        data = _load(account)
        data[_consumed_key(stage_code, tier)] = True
        _save(account, data)
        return PITY_HP_MULT
    return 1.0


def apply_battle_end(account: Account, stage_code: str, tier: StageDifficulty | str, won: bool) -> None:
    """Called after the final outcome is known. Mutates pity state per spec:
      WIN → clear both keys for this (stage, tier).
      LOSS w/ consumed=True → reset count to 0, clear _consumed.
      LOSS w/ consumed=False → increment count by 1.
    """
    data = _load(account)
    key = _key(stage_code, tier)
    ckey = _consumed_key(stage_code, tier)

    if won:
        data.pop(key, None)
        data.pop(ckey, None)
    else:
        consumed = bool(data.get(ckey, False))
        if consumed:
            # Discount didn't save them — reset cycle.
            data[key] = 0
            data.pop(ckey, None)
            # Clean up zero entries so the JSON stays small.
            if data.get(key) == 0:
                data.pop(key, None)
        else:
            current = int(data.get(key, 0))
            data[key] = current + 1

    _save(account, data)
```

```python
def test_battle_after_three_losses_gets_hp_discount(client, db_session):
    """After 3 consecutive losses on a (stage, tier), the 4th attempt fights enemies
    with 0.9x HP; on WIN the pity state clears entirely."""
    from sqlalchemy import select

    from app.models import Account, HeroInstance, HeroTemplate, Rarity, Stage, StageDifficulty
    from app.economy import mark_cleared
    from app.auth_tokens import issue_token  # adapt path if different
    from app.fail_pity import read_pity

    acc = Account(email="pity_e2e@example.com", password_hash="x")
    db_session.add(acc)
    db_session.flush()

    # Use a NORMAL stage to avoid power-floor / unlock chain.
    normal = db_session.scalar(
        select(Stage).where(Stage.difficulty_tier == StageDifficulty.NORMAL).limit(1)
    )
    assert normal is not None

    # Give the account 3 minimal-power heroes that will reliably LOSE.
    common_tmpl = db_session.scalar(
        select(HeroTemplate).where(HeroTemplate.rarity == Rarity.COMMON).limit(1)
    )
    assert common_tmpl is not None
    hero_ids: list[int] = []
    for _ in range(3):
        hi = HeroInstance(account_id=acc.id, template_id=common_tmpl.id, level=1, xp=0)
        db_session.add(hi)
        db_session.flush()
        hero_ids.append(hi.id)
    db_session.commit()

    # Manually set pity count to 3 — easier than running 3 real losing battles.
    import json
    acc.stage_pity_json = json.dumps({f"{normal.code}:NORMAL": 3})
    db_session.commit()

    # Verify the start-of-battle helper would return discount mult.
    from app.fail_pity import apply_battle_start, PITY_HP_MULT
    # Re-read account to get a fresh row (the upcoming endpoint call will run start logic itself).
    db_session.refresh(acc)

    # Smoke-confirm the helper sees count=3.
    count, consumed = read_pity(acc, normal.code, StageDifficulty.NORMAL)
    assert count == 3
    assert consumed is False

    # End-to-end: trigger one battle. The endpoint should apply the discount
    # internally; we don't assert specific HP values (depends on RNG + level scaling),
    # we assert pity state changes.
    token = issue_token(acc.id, acc.token_version)
    r = client.post(
        "/battles",
        headers={"Authorization": f"Bearer {token}"},
        json={"stage_id": normal.id, "team": hero_ids},
    )
    assert r.status_code == 201, r.text

    # After the battle, pity state must have advanced. Re-read.
    db_session.refresh(acc)
    count_after, consumed_after = read_pity(acc, normal.code, StageDifficulty.NORMAL)

    if r.json()["outcome"] == "WIN":
        # Discount fired and team won — both keys cleared.
        assert count_after == 0
        assert consumed_after is False
    else:
        # Discount fired and team still lost — count reset to 0, consumed cleared.
        assert count_after == 0
        assert consumed_after is False
```
